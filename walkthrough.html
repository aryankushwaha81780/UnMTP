Hybrid File Transfer System

A two-phase, client-server file transfer tool built in Python. Combines zero-copy kernel streaming for massive files (≥5 GB) with tar-pipe streaming for everything else — designed for high-speed LAN transfers between Android/Termux and Windows.

Architecture & Internal Functionality

The system follows a client-server model using raw TCP sockets. The host (server) runs on the source device (Android/Termux), recursively scans a target folder, and the pull (client) runs on the destination machine (Windows PC). A custom binary protocol with single-byte control flags orchestrates two distinct transfer phases.

SERVER hybrid-host.py

Runs on the source device. Scans the target directory tree, identifies files exceeding the 5 GB threshold, then serves the entire folder over a single TCP connection. Zero-copy kernel sendfile() is used for massive files, while the rest is streamed as a live tar archive.

CLIENT hybrid-pull.py

Runs on the destination machine. Auto-detects the Wi-Fi gateway IP, connects to the host server, and listens for control flags. Receives big files individually, then extracts the tar stream for everything else. Includes a robust retry loop with user prompts.

Wire Protocol Specification

All communication happens over a single persistent TCP connection on port 5001. The protocol is binary, big-endian, and uses single-byte control flags to delineate phases.

Phase 1 — Zero-Copy File Transfer

Flag 0x01

Repeated for each file ≥ 5 GB. Sent via socket.sendfile() for kernel-level zero-copy.

Sent exactly once. A streaming tar archive containing all files below the threshold, preserving directory structure.

Host Module Deep Dive

hybrid-host.py runs on the source device (Android/Termux). It implements a single-client TCP server with a two-phase transfer pipeline.

Phase 1 — Zero-Copy Kernel Transfer

Files ≥ 5 GB are sent individually using socket.sendfile(), which leverages the OS kernel's sendfile(2) syscall to transfer data directly from file descriptor to socket — without copying through userspace. This dramatically reduces CPU usage and memory pressure for multi-gigabyte files.

Phase 2 — Streaming Tar Archive

All remaining files plus the full directory structure are bundled into a streaming tar archive (mode='w|'). A custom tar_filter() excludes files already transferred in Phase 1. The tar is written directly to the socket's file wrapper with no intermediate disk I/O.

Key Design Decisions

  • Send buffer set to 8 MB (SO_SNDBUF = 8388608) for sustained throughput
  • 5 GB threshold balances tar overhead vs. zero-copy benefits
  • Relative paths computed from os.path.dirname(base_dir) — preserving root folder name
  • Single connection model — no multiplexing or parallel streams
  • SO_REUSEADDR enabled for fast server restart without TIME_WAIT issues

Pull Module Deep Dive

hybrid-pull.py runs on the destination Windows machine — the more complex script, featuring auto network discovery, validated input, a connection retry loop, and dual-phase receive logic.

NET Gateway Auto-Detection

Two-tier strategy: primary uses PowerShell Get-NetRoute with 3s timeout. Fallback parses ipconfig output prioritizing Wi-Fi, then any adapter with IPv4. All candidates validated against strict IPv4 regex.

I/O Destination Validation

Creates directory tree via os.makedirs, tests writability by creating and deleting a temp file (.write_test_temp), loops until valid writable path or user cancels.

LOOP Connection Retry Logic

ConnectionRefused/Timeout: resets IP, re-prompts. ConnectionReset/Aborted: asks user y/n. KeyboardInterrupt: graceful exit. First attempt uses auto-detected IP; retries prompt with previous default.

BUF Buffer Configuration

Socket receive buffer: 8 MB. Python file-object buffering: 8 MB. Phase 1 write chunks: 1 MB. Socket timeout: 60 seconds.

NOTE Inter-Script Relationship

The two scripts are completely independent — they don't import each other and share no module. They communicate exclusively over TCP at runtime. The only contract between them is the wire protocol: control flags, header format, and tar streaming mode. Either can be reimplemented in any language.

Security & Limitations

⚠️ No Encryption

Raw bytes over unencrypted TCP. Suitable for trusted LANs only.

⚠️ No Authentication

Server accepts any incoming connection without credentials.

⚠️ Single Client

listen(1) — accepts exactly one connection. No concurrency.

⚠️ Tar Traversal

extractall() without filter in older Python could allow path traversal.

✅ LAN Design

Explicitly designed for local Wi-Fi. Gateway detection and Termux paths reflect this.

✅ Zero Dependencies

No third-party packages — no supply-chain risk. Pure stdlib.
